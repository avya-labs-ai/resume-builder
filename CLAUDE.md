# CLAUDE.md — Resume Builder

Context for any Claude Code session working in this repo.

---

## What this project is

A single-purpose automation: take a job description, produce a tailored CV and cover letter in **English and German**, as compilable `.tex` files, organized into a per-application output folder.

There is no application server, no API, no Python runtime. The engine is Claude Code itself, invoked via the `/apply-for-job` slash command.

---

## How it works

Run `/apply-for-job` in a Claude Code session. The command (defined in [.claude/commands/apply-for-job.md](.claude/commands/apply-for-job.md)) will:

1. Read [input/resume.tex](input/resume.tex) — the LaTeX CV template
2. Read [input/profile.md](input/profile.md) — the consolidated personal/career profile
3. Read all files in [proj_refs/](proj_refs/) — active project summaries (see below)
4. Take the JD from `$ARGUMENTS` or prompt you to paste it
5. Print a gap analysis to the chat (terminal-only, never saved)
6. Derive a folder slug `{company}-{role}` from the JD
7. Write 4 LaTeX files into `output/{slug}/`:
   - `cv_en.tex`, `cv_de.tex`, `cover_letter_en.tex`, `cover_letter_de.tex`

## Input files

### `input/resume.tex` — LaTeX CV template
The master CV source. Preserved as a LaTeX template — the slash command keeps its `\documentclass`, packages, and section structure intact, adapting only content.
**Do not modify this file during a CV generation run.**

### `input/profile.md` — Consolidated profile
The single source of truth for personal data, career history, skills, framing rules, and constraints. Merged from both `girish_profile_summary.md` and the earlier `profile.md`. Includes:
- Identity / framing rules (Founder-first vs. Consultant framing)
- Full career history with bullet-point detail per role
- Skills, toolchain, standards
- Achievements, domain expertise, career goals, constraints
- Contact details, education, languages

**Update this file** when roles, skills, or framing preferences change.

### `proj_refs/` — Active project summaries
One Markdown file per active project, generated by the `/project-summary` command in the Avya AIOS. These give the CV agent **deep engineering context** about what is being built right now:

| File | Project |
|------|---------|
| `avya-labs-website_summary_17_05.md` | avya-labs.com marketing site (Vercel, VAPI, Cal.com, GDPR-compliant) |
| `project_summary_for_cv.md` | Avya Bookkeeping Agent (PRD-first, 12 ADRs, formal test plan, WAT framework) |
| `project_summary_17_05_2026.md` | Avya AIOS — personal AI Operating System scaffold |
| `weavy_skill_summary_17_05_2026.md` | Weavy Workflow Agent — Claude Code skill for generative AI workflow JSON |

**When drafting a CV**, the agent reads all `proj_refs/` files and selects the most JD-relevant project(s) to feature or reference. Project detail from `proj_refs/` supplements — and takes precedence over — the high-level project summaries in `input/profile.md`.

**Update `proj_refs/`** by running `/project-summary` in the relevant Avya AIOS session and dropping the output file here.

## Working rules

- **Never modify [input/resume.tex](input/resume.tex) or [input/profile.md](input/profile.md)** during a CV run — they are source files.
- **All generated output belongs under `output/{slug}/`** — never write CVs/letters anywhere else.
- **Preserve the LaTeX template** from `input/resume.tex` — same `\documentclass`, same packages, same section structure. Adapt content only.
- **Identity framing:** Default to "Founder, Avya Labs." Downshift to "Independent AI Consultant — Avya Labs" only for JDs where founder status is a flight-risk signal (Big-4 Senior Consultant tracks, large-enterprise in-house roles). Never invent external paying consulting income.
- **German is not literal translation.** Use natural professional German phrasing, localized section names (Berufserfahrung, Ausbildung, Kenntnisse), German date formats (`DD.MM.YYYY`), and German business-letter conventions (`Sehr geehrte Damen und Herren,` … `Mit freundlichen Grüßen,`).
- **Be truthful.** Adaptation = emphasize, reorder, reframe. Never invent experience or skills.
- **One job description = one output subfolder.** Re-running with the same JD overwrites the previous output.
- **CVs must fit on a single A4 page.** Summary: 3–4 lines. Skills: 4–6 grouped bullets with ` | ` separators. Most-relevant role: ≤4 bullets. Mid-career roles: 2–3 bullets. Older roles (>7 years): 1 line. No Hobbies section unless JD signals cultural fit. Font: `\fontsize{10pt}{12pt}\selectfont` — do not shrink further; drop lowest-relevance bullets first.
- **ATS-friendly always.** No math-mode glyphs (`$\diamond$` etc.) — use ` | ` separators. Spell out URLs (no hidden `\href` link text). Standard section names (`Skills`, `Experience`, `Education`, `Languages` in EN; `Kenntnisse`, `Berufserfahrung`, `Ausbildung`, `Sprachen` in DE). Plain ASCII hyphens in date ranges. No tables / multi-column / images for ATS-critical content. Include exact JD keywords verbatim where truthful. Full rules in the slash command.
- **NEVER** use em dashes. When in need always use '-'

## Editing the slash command

If the workflow needs to change (add a language, change output filenames, adjust the ATS or 1-page rules), edit [.claude/commands/apply-for-job.md](.claude/commands/apply-for-job.md). That single file defines the entire automation.

## Out of scope

- LaTeX compilation (compile in your own editor)
- Job-board scraping / API integration
- Tracking applications, deadlines, status
- Email / cover-letter delivery

This project ends at "4 `.tex` files in a folder."
