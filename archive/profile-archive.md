### 1. Avya Labs Marketing Website with Integrated Lead Qualification AI Voice Agent — `avya-labs.com`
Production marketing site engineered as the single sales funnel converting visitors into Cal.com-booked discovery calls, with end-to-end legal compliance for the German/EU market. Its one job: move a qualified visitor from "just landed" to "call booked" — with a legally defensible record of how their data was handled along the way.

- **Stack:** Pure HTML/CSS/vanilla JS (no framework), Tailwind v3, Three.js, Vercel hosting + auto-deploy from `main` + serverless functions, Cal.com REST API + embed, VAPI, n8n + Google Sheets CRM, self-hosted Cormorant Garamond + Inter fonts
- **Engineering highlights:**
  - Two-lane post-booking architecture — server lane (CRM persistence + VAPI trigger) and browser lane (UX overlay) share only the booking `uid`; neither depends on the other completing first
  - Branded post-booking UX flow — booked visitors get a personalised confirmation overlay (their name) plus an immediate VAPI call-back offer; a graceful fallback overlay fires on any API failure so the experience never breaks
  - Same-origin Vercel proxies for every external call (Cal.com, n8n) — browser never sees API keys or third-party URLs; satisfies strict CSP without `unsafe-*` carve-outs and keeps all credentials server-side
  - CSP with SHA-256 hash-whitelisted inline scripts (no blanket `unsafe-inline` for `script-src`)
  - Resolved a production race condition by switching from n8n CRM mirror reads to direct Cal.com REST API reads via same-origin proxy (April 2026 cutover) — the mirror raced the populating webhook and surfaced the wrong name/overlay to every visitor
  - Fully self-hosted assets (fonts, Three.js, Tailwind) to eliminate US-CDN data-transfer risk under GDPR
  - Test helpers gated to localhost — `window.__testCalBooking*` globals attach only on `localhost`/`127.0.0.1`; the production bundle exposes zero internals
  - No build framework — Tailwind is the only build step; keeps the deploy surface minimal and eliminates framework churn
- **Compliance:** GDPR & TTDSG — privacy policy discloses Cal.com and VAPI as Art. 28 processors; German legal Impressum per TMG §5; HSTS, X-Frame-Options, all standard security headers configured in `vercel.json`
- **AI-assisted dev & continuity:** Built using Claude Code under strict project rules captured in `CLAUDE.md`; architectural state preserved in a `context-handoff.md` so a fresh agent or developer can resume across sessions; Puppeteer-based screenshot helper for visual QA

Production-grade AI agent system that automates 5 agile ceremony types across a full OKR-to-task cascade hierarchy for Avya Labs — replacing the accountability, planning, and retrospective functions of an ops team that does not exist. Sole-authored in 6 weeks (May–June 2026) using strict TDD throughout; 305+ tests passing with no live API required for the default run.

- **Stack:** Python 3.11+, Pydantic v2, pytest + Hypothesis (property-based testing), httpx, mypy --strict, ruff, uv, Notion REST API + ntn CLI, Claude Code (Anthropic) as primary development tool, argparse CLI, circuit breaker pattern, PEP 735 dependency groups with committed lockfile
- **Engineering highlights:**
  - **305+ automated tests** across 5 phases; fully offline by default — integration tests gated behind an environment flag registered in both `pyproject.toml` and `conftest.py` (defense-in-depth so there is no single point of failure that silently converts integration tests into unit tests)
  - **Dual-transport Notion facade** — routes calls to either `ntn` CLI (primary) or `httpx` REST (fallback) through a single `\_route()` helper; transport injected at construction time so ceremony handlers never reference transport directly and unit tests swap it with zero other code changes
  - **Circuit breaker with mandatory auth-error bypass** — auth failures must not trip the breaker because both transports share the same credentials; tripping on auth would silently route to REST, which also fails, masking the real problem; recognising this edge case is the seniority signal
  - **Notion silent-failure problem solved** — Notion never returns 403; an inaccessible database returns an identical empty result to a legitimately empty one; solved via known-page-ID cross-check in `verify-setup` (9-point health check CLI); without this, broken workspaces pass silently
  - **Graph termination proved with property-based testing** — used Hypothesis to generate arbitrary cascade graph inputs (including circular Notion relations) and prove `CascadeGraph` always terminates within `MAX\_DEPTH=5`; parametrized unit tests only prove termination on cases you thought to write
  - **`NtnRunner` as single subprocess boundary** — all ANSI stripping, JSON slicing, timeout handling, and typed exception hierarchy (`NtnAuthError`, `NtnApiError`, `NtnNotInstalledError`) in one place; no raw `subprocess.run` anywhere else; pre-compiled regex at module load
  - **Risk-ordered build sequence** — Notion integration layer built first (not the most visible feature) because it was the highest-risk dependency; fragile plumbing would make every ceremony built on top fragile
  - **Type-based Notion schema discovery** — `get\_title()` finds the property with type `title` rather than a hardcoded key; survives workspace renames without code updates; cached to `.planning/notion-schema.json`
  - **Pydantic two-tier validation** — strict mode on internal contracts (`SkillOutput`), lenient `extra="allow"` on Notion response models; resilient to Notion adding undocumented properties without versioning guarantees
  - **AI-native engineering discipline** — all 21 completed plans executed with Claude Code under human-directed TDD; AI output that failed `mypy --strict` or `pytest` gates was rejected and corrected; demonstrates senior-level ability to specify correctness before a line is written and enforce it on generated output
